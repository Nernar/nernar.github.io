    API Reference

    Namespaces

    NamespaceDescription
    ActorRenderer
    AddonEntityRegistryModule used to manage custom entities added via add-ons
    AnimationAnimations are used to display some 3d models in the world without use of entities
    ArmorModule used to manage armor's behavior
    BlockModule used to create and manipulate blocks. The difference between terms "block" and "tile" is in its usage: blocks are used in the inventory, tiles are placed in the world and have different ids for some vanilla blocks. Use [[Block.convertBlockToItemId]] and [[Block.convertItemToBlockId]]
    BlockRendererModule used to create blocks with any custom model
    CallbackModule used to handle callbacks. See for details about the callback mechanism and the list of pre-defined callbacks
    com
    CommandsNamespace used to manipulate minecraft commands
    Config
    CustomEnchantModule to create custom enchantments.
    DebugDefines some useful methods for debugging
    DimensionsNamespace used to create and manipulate custom dimensions
    EntityModule used to manipulate entities (mobs, drop, arrows, etc.) in the world. Every entity has its unique numeric id which is often used across this module as the first function parameter
    EntityAIA set of predefined entity AI types
    EntityAIClass
    FileToolsModule that provides methods to work with android file system
    GameModule that provides some general game-related functions
    GameObjectRegistry
    GenerationUtilsModule used to simplify generation tasks in mods logic
    ICRenderClass used to define block models that depend on surrounding blocks. Some examples of such blocks are wires, pipes, block structure parts, etc.
    IDDataModule containing [[ItemID]] and [[BlockID]] values
    IDRegistryModule used to manage item and block ids. Items and blocks have the same underlying nature, so their ids are interchangeable. Though, the blocks are defined "twice", as an item (in player's hand or inventory) and as a tile (a block placed in the world)
    ItemModule used to define items and their properties
    ItemModelNamespace used to change item models in player's hand and/or inventory. By default, if the block has an [[ICRender]], it is automatically applied as item's model
    LiquidRegistry
    LoggerModule used to log messages to Inner Core log and android log
    MobRegistry
    MobSpawnRegistry
    Mod
    ModAPIModule used to share mods' APIs
    ModPack
    NativeModule containing enums that can make user code more readable
    NBTNBT (Named Binary Tag) is a tag based binary format designed to carry large amounts of binary data with smaller amounts of additional data. You can get or set nbt tags of [[Entity]] (entities), [[NativeTileEntity]] (native tile entities, such as chests or beacons) and [[ItemExtraData]] (items). To get more information about these data structures, see [this page](http://web.archive.org/web/20110723210920/http://www.minecraft.net/docs/NBT.txt)
    NetworkNew module to work with client and server packets in multiplayer.
    ParticlesModule to work with vanilla and custom particles
    PlayerModule used to manipulate player. Player is also an entity in Minecraft, so you can use all the functions from [[Entity]] module as well. To get player's entity id, call [[Player.get]] function
    RecipesModule used to manipulate crafting recipes for vanilla and custom workbenches
    Render
    RenderMesh
    SaverModule used to save data between world sessions
    ThreadingModule used to create and manipulate threads. Threads let you execute time-consuming tasks without blocking current execution thread
    TileEntityTileEntity is a powerful mechanism that allows for creation of interactive blocks such as chests, furnaces, etc.
    ToolAPIModule used to manage block and tools material and create tools with all required properties
    TranslationModule that can be used to localize mods All default strings (e.g. item names, windows titles, etc.) in the mod should be in English. Add translations to these strings using [[Translation.addTranslation]]. For items and blocks translations are applied automatically. For the other strings, use [[Translation.translate]]
    UI
    UpdatableModule used to create and manage Updatables. Updatables provide the proper way to manage objects that update their state every tick. Updatables may not be notified every tick, if there are too many, to avoid user interface freezes
    WorldModule that allows to work with current Minecraft world Most of the methods are out of date in multiplayer, use BlockSource instead

    Classes

    ClassDescription
    ActorRendererClass, upon which armor and attachments render is based It is a model that consists of parts, same as in deprecated [[Render]], but more abstract, allows creating root parts instead of inheritance from old humanoid model
    AttachableRenderClass used to attach attachables to entities
    BlockSourceNew class to work with world instead of some methods from [[World]] module.
    BlockStateClass to work with vanilla blocks parameters
    ConfigJson configuration file reading/writing utility
    CustomBiomeClass used to create custom biomes. Note that Minecraft has a limit of 256 biomes and there are already more than 100 vanilla biomes, so do not overuse this functionality. See
    CustomEntity
    EntityAIClassClass used to create new entity AI types
    EntityModel
    GameObjectClass used to create and manipulate game objects. Game objects are [[Updatable]]s that are being saved between game launches
    ItemContainerNew type of TileEntity container that supports multiplayer
    ItemContainerSlot
    ItemExtraDataClass representing item extra data. Used to store additional information about item other then just item id and data
    NetworkClientClass that represents network client
    NetworkConnectedClientListClass to work with definite couple of clients, bound by certain conditions
    NetworkEntityClass that represents network entity of the block, currently is not learned
    NetworkEntityTypeClass that represents network entity type
    PlayerActorClass to manipulate with separate players. It is temporary! It exists only 1 server tick!
    RenderClass that is used to give mobs, animations and blocks custom shape.
    RenderMeshClass representing a set of vertices with some other parameters required to display them correctly. Used as block, entity and item models, in animations and actually anywhere you need some physical shape
    ShaderUniformSet
    SyncedNetworkDataClass to work with values, synchronized between server and all clients
    TextureClass representing texture that can be animated

    Enumerations

    EnumerationDescription
    EArmorTypeDefines armor type and armor slot index in player's inventory
    EBlockRenderLayerDefines possible render layers (display methods) for blocks
    EBlockSideDefines numeric representation for each block side
    EBlockStatesDefines numeric representation for each vanilla block state
    EColorDefines text colors and font styles for chat and tip messages
    EDimensionDefines numeric representation for three vanilla dimensions
    EEnchantmentDefines numeric ids of all vanilla enchantments
    EEnchantTypeDefines what enchantments can or cannot be applied to every instrument type
    EEntityTypeDefines all vanilla entity type numeric ids
    EGameDifficultyDefines possible game difficulties
    EGameModeDefines possible game modes
    EItemAnimationDefines item animation types
    EItemCategoryDefines vanilla item categories in creative inventory
    EMobRenderTypeDefines vanilla mob render types
    ENbtDataTypeDefines numeric representation for each NBT data type
    EParticleTypeDefines all existing vanilla particles
    EPlayerAbilityDefines player's abilities. See for details
    EPotionEffectDefines vanilla potion effects
    ETileEntityTypeDefines numeric representation for vanilla TileEntity types. Use [[NativeTileEntity]] class to work with them.
    VanillaBlockIDNumeric IDs of vanilla blocks in the inventory
    VanillaItemIDNumeric IDs of vanilla items
    VanillaTileIDNumeric IDs of vanilla blocks placed in the world

    Variables

    VariableDescription
    __config__Main mod configuration manager, settings are stored in config.json file. For more information about config.json, see
    __dir__Full path to the mod's directory, ends with "/"
    __mod__Java object of the mod, contains some useful values and methods
    __modpack__Full path to current Inner Core modpack directory
    __name__Mod name
    __packdir__Full path to current Horizon pack directory
    BlockIDObject containing custom block string ids as keys and their numeric ids as values
    ItemIDObject containing custom item string ids as keys and their numeric ids as values

    Functions

    FunctionDescription
    __debug_typecheck__()Empty function used to verify Rhino functionality. Should not be called by hand
    alert(arg)Displays android.widget.Toast with specified message. If this function is called more then once, messages are stacked and displayed together
    ConfigureMultiplayer(args)Function that must be written in launcher.js to enable multiplayer configuration
    EntityAIWatcher(customPrototype)
    EntityModelWatcher(entity, model)
    EXPORT(name, lib)Exports object from library using specified name
    getCoreAPILevel()
    getMCPEVersion()
    IMPORT_NATIVE(name, target)Injects methods from C++ into the target object to use in the mod
    IMPORT(name, value)Imports library dependency. Libraries should be stored in the *"libraryDir"* directory, specified in your *build.config*. You can either import the whole library or single function/value using value parameter
    importLib(name, value)Same as [[IMPORT]], consider using [[IMPORT]] instead
    LIBRARY(description)Library declaration, specifies all the information about library it is called from. Cannot be called from user code.
    runCustomSource(name, scope)Runs custom source in the specified context by its name. Define custom sources using *"sourceType": "custom"* for the source in your *build.config*.
    runOnClientThread(func)Runs specified function in the client thread. Same as [[runOnMainThread]], but for the client side.
    runOnMainThread(func)Runs specified function in the main thread
    WRAP_JAVA(name)Allows to create new JS modules imported from Java code and use it in the mod
    WRAP_NATIVE(name)Allows to create new JS modules imported from C++ code and use it in the mod

    Interfaces

    InterfaceDescription
    BlockPositionObject representing coordinate set with side data
    ColorObject representing RGB color
    GameObjectPrototype
    ItemInstanceObject representing item instance in the inventory
    ItemModelClass representing item model in player's hand and/or inventory. To get an instance of this class from your code, use [[ItemModel.getFor]] static function. The coordinates of the full block in player's hand or inventory is (0, 0, 0), (1, 1, 1), so it is generally recommended to use the models that fit that bound at least for the inventory
    LookAngleObject representing pitch/yaw angle set (in radians)
    NativeTileEntityInterface providing access to native tile entities - chests, hoppers, furnaces, smelters, etc. See full lists of supported native tile entities in the [[Native.TileEntityType]] enum
    TileObject representing block in the world
    TileEntity
    UpdatableUpdatable is an object that is notified every tick via its [[Updatable.update]] method call
    VectorObject representing the set of coordinates in the three-dimensional world
    WeatherObject representing current weather in the world

    Type AliasDescription
    ArmorTypeString types of armor to be specified when calling [[Item.createArmorItem]]
    DefaultRenderTemplateDefault render templates used inside of Inner Core, currently there are only default armor models
    ItemInstanceArrayArray of three or four elements representing item id, count, data and extra respectively. Uses in block drop functions
    jbyteType used to mark Java bytes
